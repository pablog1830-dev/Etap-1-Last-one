# 2A v1.1 — Execution Specification

## Status
This document is the binding implementation scope for **2A v1.1**.
It supersedes exploratory Etap-1 style refactors for the execution layer.

## Objective
2A v1.1 is the first narrowed execution stage for **crypto scalping**.
The goal is **not** to build the full target architecture.
The goal is to verify whether the current core produces a **fast, executable decision** and preserves **net edge after costs**.

## Scope
- Symbols: **BTCUSDT**, **ETHUSDT**
- Market types: **spot**, **linear**
- Each symbol and market type must be evaluated **separately**
- Timeframe profile:
  - `context_tf = 15m`
  - `structure_tf = 5m`
  - `execution_tf = 1m`
- Canonical time: **UTC**
- Canonical base data: **1m OHLCV**
- `5m` and `15m` must be built **locally from 1m**

## Explicitly out of scope for 2A
Do **not** implement or expand:
- Prediction
- ML
- thesis/survival labels
- full `FVGObject`
- full `RegimeState`
- full `SurvivalState`
- mark/index/premium
- funding
- open_interest
- account_ratio

These remain the core and must **not** be logically rebuilt:

## Data integrity mini
Mandatory fields:
- `data_quality_score`
- `gap_flag`
- `partial_history_flag`
- `bar_alignment_ok`
- `degradedModeState`

Behavior:
- `signal_suppression` is a **hard blocker**
- `reduced_confidence` reduces quality and biases toward `DELAY`
- state is **three-valued**, not binary

## Quality decomposition
Mandatory outputs, each in `0..100`:
- `thesis_quality`
- `location_quality`
- `execution_quality`

These must be derived from the existing core, not from a rebuilt architecture.

## Setup quality
Formula is fixed:

```text
setup_quality = max(0, min(thesis_quality, location_quality, execution_quality) - penalties_total)
```

## Hard blockers
- `signal_suppression`
- `hard_structural_invalidation`
- `no_valid_sl_tp_geometry`
- `expired_setup_window`
- `profile_disallows_entry`

## Liquidity state rules
`BASIC_LIQUIDITY_STATE` must resolve with minimal precedence:
- `hard_invalidation` -> `CANCEL`
- `reclaim_needed` -> `WAIT_RECLAIM`
- `sweep_context` -> usually `DELAY`
- `none` -> no direct effect

Important:
- `reclaim` and `WAIT_RECLAIM` work **only** when `basicLiquidityState = reclaim_needed`
- they do **not** activate on raw `sweep_context`

## Entry mode rules
`entryMode` must be resolved from:
- market profile
- market type
- setup quality
- penalties
- liquidity state
- degraded mode

No discretionary/manual mode switching.

#### entryMode
`aggressive`

- `execution_quality >= 68`
- `setup_quality >= 60`

- `degradedModeState = none`

`standard`
- `thesis_quality >= 62`
- `location_quality >= 58`
- `execution_quality >= 58`
- `setup_quality >= 55`
- `penalties_total <= 16`
- `basicLiquidityState in {none, sweep_context}`
- `degradedModeState != signal_suppression`

`reclaim`
- `basicLiquidityState = reclaim_needed`
- reclaim within `<= 3` closed `1m` bars
- after reclaim:
  - `thesis_quality >= 60`
  - `location_quality >= 60`
  - `execution_quality >= 58`
  - `setup_quality >= 54`
  - no hard blockers

`disabled`
- every other case

### BTC WAIT_RECLAIM

- `location_quality >= 58`

## Acceptance criteria
### Technical
- compilation without errors
- no undeclared identifiers
- no look-ahead
- one decision on closed `1m` bar
- no unstable intrabar transitions
- parity across `CANONICAL`, `FULL_COPY`, `TV_LIVE`

### Trading
Evaluate separately for each symbol and each market type.
One market cannot mask weakness of another.

Targets:
- `setup_frequency_day >= 1`
- `setup_frequency_week >= 5`
- `median_time_to_entry <= 2` bars `1m`
- `p90_time_to_entry <= 4` bars `1m`
- `median_hold_time <= 8` bars `1m`
- `p90_hold_time <= 15` bars `1m`
- `cancelled_opportunity_rate <= 0.50`
- `missed_move_ratio <= 0.40`
- `net_expectancy_after_fees >= 0` or explicitly documented tolerance
